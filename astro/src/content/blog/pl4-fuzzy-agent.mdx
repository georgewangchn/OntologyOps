> 副标题：当症状不再是有/无 —— 模糊推理如何让 Agent 理解「有多严重」

## 一、引言：P4 很准，但用户更不会用了

P4 引入了第四种推理范式——**模糊推理**。症状不再是"有/无"（二元），而是"严重度 0-1"（连续）。疾病不再是"确诊/排除"（二元），而是"置信度 0-1"（连续）。

但 P4 的使用门槛比 P1-P3 都高：

- 用户要传 `symptom_details`，记录每个症状的严重度详情
- 发热要传 `{"value": 39.5}`（体温值）
- 呕吐要传 `{"frequency": "频繁"}`（频率）
- 腹泻要传 `{"type": "水样", "color": "暗红"}`（类型+颜色）
- 不传 details 就退化为基线严重度，推理质量下降

**PL4 的核心思路**：

> 让 LLM Agent 充当「模糊化前端」——用户说"猫体温 39.5°C、呕吐了四五次、拉的是水样带血的便便"，Agent 自动提取严重度详情、触发 Mamdani 模糊推理，再把连续置信度翻译成人类可读的诊断报告。

PL1-PL3 的 Agent 主要做"翻译"——把自然语言翻译成结构化输入。PL4 的 Agent 多了一个职责：**模糊化**——把自然语言中的程度描述转换为连续严重度值。

## 二、PL4 的独特挑战：Agent 要追问「有多严重」

### 2.1 PL1-PL3 vs PL4 的对话策略差异

**PL1-PL3 的对话**：
```
用户：我家猫发烧了，还吐了
Agent：好的，记录发热和呕吐。还有其他症状吗？
用户：还拉肚子了
Agent：记录腹泻。信息够了，开始推理。
```

**PL4 的对话**：
```
用户：我家猫发烧了，还吐了
Agent：好的，记录发热和呕吐。请问：
  - 发热体温是多少？（38.5°C 和 40°C 的严重度不同）
  - 呕吐频率？（偶尔/多次/频繁）
用户：39.8°C，吐了四五次了
Agent：记录：发热(39.8°C, 严重度0.80) + 呕吐(频繁, 严重度0.90)
  还有其他症状吗？
用户：还拉肚子了，水样的，带血
Agent：记录：腹泻(水样+暗红, 严重度0.90)
  信息够了，开始模糊推理。
```

PL4 的 Agent 必须追问严重度详情——因为 39.8°C 的高烧和 38.5°C 的低烧，在模糊推理中是完全不同的输入。

### 2.2 系统提示词中的追问策略

PL4 的系统提示词明确要求 Agent 追问严重度：

```
## 对话流程
1. 问候用户，询问宠物物种和症状
2. **关键步骤**：对每个症状追问严重度详情：
   - 发热 → 追问体温值（°C）
   - 呕吐 → 追问频率（偶尔/多次/频繁）
   - 腹泻 → 追问类型（成型/软便/水样）和颜色（正常/暗红/血）
3. 信息收集完毕后，调用 run_fuzzy_reasoning 输出诊断报告
```

## 三、PL4 的八个工具

| # | 工具名 | 功能 | PL4 特有 |
|---|--------|------|----------|
| 1 | `lookup_symptom_fuzzy` | 查找症状 + 基线严重度 | ✅ 返回基线严重度 |
| 2 | `lookup_disease_fuzzy` | 查找疾病详情 | |
| 3 | `add_observation` | 记录症状（含严重度详情） | ✅ details 影响推理 |
| 4 | `set_pet_info` | 设置宠物信息 | |
| 5 | `run_fuzzy_reasoning` | 运行 Mamdani 模糊推理 | ✅ 三维输入 |
| 6 | `explain_fuzzy_reasoning` | 解释推理链 | ✅ 覆盖率/强度/排除度 |
| 7 | `get_symptom_severity` | 查询症状模糊化严重度 | ✅ PL4 独有工具 |
| 8 | `get_case_summary` | 病例摘要（含严重度） | ✅ 显示 details |

### 3.1 get_symptom_severity：PL4 独有工具

PL4 多了一个 PL1-PL3 都没有的工具——`get_symptom_severity`。这个工具让用户看到症状如何从自然语言映射为连续严重度：

```
用户：我之前说猫发热 39.8°C，这个在推理中是怎么用的？

Agent（Act）：调用 get_symptom_severity("发热")

Agent：
  症状：发热
  当前严重度：0.80
  基线严重度：0.80
  详情：value=39.8

  严重度映射规则：
    体温 ≥ 40.0°C → 严重度 1.0
    体温 ≥ 39.5°C → 严重度 0.80
    体温 ≥ 39.0°C → 严重度 0.60
    体温 < 39.0°C → 严重度 0.30
```

这个工具让模糊推理的"黑盒"变成了"白盒"——用户可以看到 39.8°C 是怎么变成 0.80 的。

## 四、explain_fuzzy_reasoning：三维推理链解释

PL1-PL3 的解释工具展示的是"匹配/未匹配症状列表"。PL4 的解释工具展示的是**三维模糊输入的计算过程**：

```
用户：为什么猫瘟的置信度是 0.55 而不是更高？

Agent（Act）：调用 explain_fuzzy_reasoning("猫瘟")

Agent：
  疾病：猫瘟

  必要症状：
    ✅ 发热（严重度=0.80）
    ✅ 呕吐（严重度=0.90）
    ✅ 腹泻（严重度=0.90）

  排除症状：
    ⚠️ 咳嗽（未命中）
    ⚠️ 流鼻涕（未命中）

  模糊推理输入：
    覆盖率：1.00（3/3 必要症状全出现）
    强度：0.87（已出现症状严重度均值）
    排除度：0.00（排除症状未出现）

  Mamdani 推理流程：
    1. 模糊化：覆盖率=高, 强度=高, 排除度=无
    2. 规则触发：覆盖率=高 ∧ 强度=高 ∧ 排除度=无 → 置信度=高
    3. 合成：所有规则输出取 max
    4. 去模糊化：重心法 → 连续置信度

  与 P1-P3 的区别：
    · 置信度是连续值（0-1），不是二元（确诊/排除）
    · 排除症状不完全排除疾病，而是降低置信度
    · 症状严重度影响推理结果（39.8°C 高烧 > 38.5°C 低烧）
```

这个解释比 PL1-PL3 更丰富——不仅告诉你"匹配了什么"，还告诉你"匹配的程度如何影响结果"。

## 五、PL4 的诊断桥接：symptom_details 传递

PL4 的 `diagnose.py` 有一个 PL1-PL3 都没有的挑战——**symptom_details 的传递**。

agent_core 的 `ConversationState.to_case_dict()` 不一定包含 `symptom_details`。PL4 需要从 `observations` 的 `details` 字段中提取：

```python
def _convert_case_dict(case_dict):
    p4_case = {}
    p4_case["pet_type"] = _SPECIES_MAP.get(raw_type, raw_type)
    p4_case["symptoms"] = case_dict.get("observations", [])
    # PL4 特有：提取 symptom_details
    p4_case["symptom_details"] = case_dict.get("symptom_details", {})
    return p4_case
```

如果 `symptom_details` 为空，P4 的推理会退化为基线严重度——所有症状的严重度都使用 `symptoms.csv` 中的默认值，丢失了用户描述的程度信息。因此 PL4 的 Agent 必须积极追问 details。

## 六、连续置信度 vs 二元结果的报告差异

PL1-PL3 的 `DiagnosisReport` 中，`level` 字段是"确诊"/"疑似"/"排除"（离散值）。

PL4 的 `level` 字段是"高"/"中"/"低"（模糊等级），对应连续置信度区间：

```python
def _defuzzify_level(confidence):
    if confidence >= 0.65:
        return "高"
    elif confidence >= 0.35:
        return "中"
    else:
        return "低"
```

PL4 的报告还多了一段**与确定性推理的区别说明**：

```
与 P1-P3 确定性推理的区别：
  · 置信度是连续值（0-1），不是二元（确诊/排除）
  · 排除症状不完全排除疾病，而是降低置信度
  · 症状严重度影响推理结果（高烧 > 低烧）
```

## 九、小结

PL4 展示了如何将 Mamdani 模糊推理包装为 LLM Agent。关键收获：

1. **Agent 的模糊化职责**：PL4 的 Agent 不仅翻译自然语言，还负责将程度描述转换为连续严重度
2. **追问策略的差异**：PL4 的 Agent 必须追问"有多严重"，而 PL1-PL3 只需要确认"有没有"
3. **连续置信度的报告**：PL4 输出"高/中/低"模糊等级 + 连续置信度，而非"确诊/排除"
4. **排除逻辑的差异**：PL4 中排除症状不完全排除疾病，而是降低置信度——更接近真实医疗诊断
5. **独有工具**：`get_symptom_severity` 让模糊推理从黑盒变白盒

PL1-PL4 覆盖了四种推理范式（OWL、Prolog、SPARQL、模糊逻辑），每种都用同一个 `agent_core` 框架，只是四个注入点不同。这验证了「核心 + 可替换领域层」设计的可扩展性。

接下来 PL5 将实现多范式融合引擎——把四种推理结果融合为最终诊断。
