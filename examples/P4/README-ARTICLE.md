## 一、引言：确定性推理的天花板

P1-P3 用三种范式解决了同一个宠物诊断问题，但它们有一个共同的假设：**症状是二元的——有或没有**。

```
P1（OWL）：  has(case, 发热) ✓  →  necessary(D001, 发热) ✓
P2（Prolog）：has(case, '发热') ✓ → forall(necessary) ✓
P3（Jena）：  :case :has :发热   →  :d001 :necessary :发热
```

但现实中的医疗诊断不是这样工作的。39.5°C 的高烧和 38.8°C 的低烧，在 P1-P3 中都被当作"有发热"——**严重程度信息被丢弃了**。

> **患者**：「医生，我的猫体温 39.5°C，呕吐了四五次，拉的是水样带血的便便。」  
> **医生**：「39.5°C 是高烧，水样带血是严重腹泻——这个组合猫瘟的可能性很大，但不能完全排除猫肠炎，因为高烧也是猫肠炎的排除指征。」

医生不会因为"有发热"就排除猫肠炎——她会根据发热的**程度**来判断。39.5°C 的高烧排除猫肠炎的力度，和 38.5°C 的低烧完全不同。

这正是《当 LLM 不够用了》第四章 4.4 节讲的核心问题：

> 确定性推理保证结论的可解释性和一致性。不确定性推理保证覆盖现实世界的灰度地带。

P4 用**模糊推理**（Mamdani 模糊逻辑）处理这个"灰度地带"——症状不再是"有/无"，而是"严重度 0-1"的连续值；疾病不再是"确诊/排除"，而是"置信度 0-1"的连续值。

## 二、技术架构：Mamdani 模糊推理

```
┌─────────────────────────────────────────────────┐
│    输入：病例（症状 + 严重度详情）               │
│    例：发热(39.5°C) 呕吐(多次) 腹泻(水样暗红)    │
└──────────────────────┬──────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│  Step 1：模糊化（Fuzzification）                │
│  症状详情 → 连续严重度（0-1）                    │
│  发热 39.5°C → 0.80，呕吐多次 → 0.70            │
│  腹泻水样暗红 → 0.90                            │
└──────────────────────┬──────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│  Step 2：计算模糊输入（双输入设计）              │
│  覆盖率 = 已出现必要症状数 / 总数                │
│  强度 = 已出现症状的严重度均值                   │
│  排除度 = 排除症状严重度最大值（模糊 OR）        │
└──────────────────────┬──────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│  Step 3：Mamdani 规则推理                        │
│  12 条 IF-THEN 规则，激活强度 = min(条件隶属度)  │
│  覆盖率=高 ∧ 强度=高 ∧ 排除度=无 → 置信度=高     │
│  覆盖率=高 ∧ 强度=高 ∧ 排除度=有 → 置信度=中     │
│  ...                                            │
└──────────────────────┬──────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────┐
│  Step 4：去模糊化（Defuzzification）             │
│  重心法（centroid）→ 连续置信度值（0-1）          │
└─────────────────────────────────────────────────┘
```

### 2.1 模糊化：从"有/无"到"有多严重"

P1-P3 的输入是症状列表 `["发热", "呕吐", "腹泻"]`——每个症状只有出现/不出现两种状态。P4 的输入增加了 `symptom_details`，记录每个症状的严重程度：

```json
{
  "symptoms": ["发热", "呕吐", "腹泻"],
  "symptom_details": {
    "发热": {"degree": "高", "value": 39.5},
    "呕吐": {"frequency": "多次", "content": "黄色液体"},
    "腹泻": {"type": "水样", "color": "暗红"}
  }
}
```

模糊化函数将这些详情映射为 0-1 的连续严重度：

```python
def compute_symptom_severity(symptom_name, case, baselines):
    # 发热：根据体温值映射
    if symptom_name == "发热" and "value" in detail:
        temp = float(detail["value"])
        if temp >= 40.0:   return 1.0   # 极高烧
        elif temp >= 39.5: return 0.8   # 高烧
        elif temp >= 39.0: return 0.6   # 中烧
        else:              return 0.3   # 低烧

    # 呕吐：根据频率映射
    if "frequency" in detail:
        if freq in ("频繁", "持续"): return 0.9
        elif freq == "多次":         return 0.7
        elif freq == "偶尔":         return 0.4

    # 腹泻：根据类型和颜色映射
    if "水样" in dtype:  base = 0.6
    if "血" in color:    base = min(1.0, base + 0.3)  # 便血加重
```

对比 P1-P3：症状只有"有(1)/无(0)"，严重度信息完全丢失。

### 2.2 隶属度函数：模糊集合的"边缘"

模糊推理的核心是**隶属度函数**——它定义了一个值"属于"某个模糊集合的程度。

P4 定义了四个模糊变量（三个输入 + 一个输出），每个用梯形隶属度函数（trapmf）描述：

**覆盖率**（必要症状的出现比例）和 **强度**（已出现症状的严重度均值）：

```
        低 [0, 0, 0.15, 0.35]      中 [0.25, 0.45, 0.55, 0.75]     高 [0.65, 0.85, 1.0, 1.0]
        ───╲                             ╱─╲                              ╱───
           ╲                           ╱   ╲                            ╱
  0  0.15  0.35  0.45  0.55  0.65  0.85  1.0
```

> 📝 **为什么用双输入？** 覆盖率回答"有几个必要症状出现"，强度回答"已出现的有多严重"。分离后可以区分"3/3 症状全在但都是低烧"（覆盖率=高, 强度=低 → 中置信度）和"1/3 症状在但极高烧"（覆盖率=低, 强度=高 → 低置信度）——医学语义完全不同。

**排除度**（排除症状的出现程度）：

```
       无 [0, 0, 0.05, 0.15]              有 [0.1, 0.3, 1.0, 1.0]
       ───╲                                 ╱───
          ╲                               ╱
 0  0.05  0.15  0.3              1.0
```

**置信度**（输出，去模糊化后得到连续值）：

```
       低 [0, 0, 0.15, 0.35]      中 [0.25, 0.45, 0.55, 0.75]     高 [0.65, 0.85, 1.0, 1.0]
```

> 📝 对比 P1-P3：不需要隶属度函数，因为症状和结论都是二元的。隶属度函数是模糊推理区别于确定性推理的核心——它让推理能够处理"边缘模糊"的问题。

### 2.3 模糊规则：Mamdani IF-THEN（三输入 12 条）

12 条规则覆盖覆盖率×强度×排除度的所有组合：

**规则矩阵（排除度=无时）：**

| 覆盖率\强度 | 低 | 中 | 高 |
|------------|-----|-----|-----|
| 低 | 低 | 低 | 低 |
| 中 | 低 | 中 | 中 |
| 高 | 中 | 中 | **高** |

**规则矩阵（排除度=有时）：**

| 覆盖率\强度 | 低 | 中 | 高 |
|------------|-----|-----|-----|
| 低 | 低 | 低 | 低 |
| 中 | 低 | 低 | 低 |
| 高 | 低 | 低 | **中** |

```python
# 核心确诊规则：全覆盖 + 高强度 + 无排除 → 高置信度
IF 覆盖率=高 AND 强度=高 AND 排除度=无 THEN 置信度=高

# 排除降级规则：全覆盖 + 高强度 + 有排除 → 中置信度（降一级，非完全排除）
IF 覆盖率=高 AND 强度=高 AND 排除度=有 THEN 置信度=中

# 缺核心症状：覆盖率低 → 低置信度（不管多严重）
IF 覆盖率=低 THEN 置信度=低

# ...共 12 条规则
```

对比 P1-P3 的规则：

| 范式 | 规则形式 | 触发方式 | 输出 |
|------|---------|---------|------|
| P1 SWRL | `has(?p,?s) ∧ necessary(?d,?s) → suspected(?p,?d)` | 二元（有/无） | 疑似（是/否） |
| P2 Prolog | `forall(necessary(D,S), has(P,S))` | 二元（全匹配） | 确诊（是/否） |
| P3 Jena | `noValue(?case :excluded ?d) → diagnosed` | 二元（无排除） | 确诊（是/否） |
| **P4 模糊** | `覆盖率=高 AND 强度=高 AND 排除度=无 → 置信度=高` | **隶属度触发** | **连续置信度** |

**关键区别**：P1-P3 的规则是"满足条件就触发"，P4 的规则是"根据隶属度部分触发"——激活强度 = min(条件隶属度)，输出是模糊集合的合成。

### 2.4 排除语义：从"一刀切"到"连续降级"

这是 P4 与 P1-P3 **最根本的行为差异**：

```
病例：猫，发热(0.8) + 呕吐(0.7) + 腹泻(0.9)
猫肠炎：necessary=[腹泻, 呕吐]  exclusion=[发热, 咳嗽]

P1-P3（确定性推理）：
  发热 ✓ → 命中猫肠炎排除症状 → 猫肠炎完全排除（不出现在结果中）

P4（模糊推理）：
  发热严重度=0.8 → 排除度=max(0.8, 0.0)=0.80
  覆盖率=2/2=1.0  强度=(0.9+0.7)/2=0.80
  规则12触发：覆盖率=高 ∧ 强度=高 ∧ 排除度=有 → 置信度=中
  去模糊化 → 0.50

  猫肠炎出现在结果中，置信度=0.50 [中]
```

**猫肠炎不被完全排除，而是置信度降为中等。** 双输入设计下，猫肠炎的覆盖率=高（必要症状全在）+强度=高（严重度都高），但排除度=有（发热命中），所以降一级为"中"而非完全压低——匹配证据强时排除只降一级。

## 三、运行示例

输入与 P1-P3 相同的病例（但增加了严重度详情）：

```python
case = {
    "pet_type": "cat",
    "symptoms": ["发热", "呕吐", "腹泻"],
    "symptom_details": {
        "发热": {"degree": "高", "value": 39.5},
        "呕吐": {"frequency": "多次"},
        "腹泻": {"type": "水样", "color": "暗红"}
    },
    "breed": "英短",
    "age": 2
}
```

输出：

```
  症状严重度（模糊化输入）：
    发热: 0.80
    呕吐: 0.70
    腹泻: 0.90

──────────────────────────────────────────────────
  📋 诊断结果（模糊推理 · Mamdani · 连续置信度）
──────────────────────────────────────────────────
  1. 猫瘟               置信度：0.86  ████████ [高]
  2. 猫肠炎              置信度：0.50  ████ [中]
  3. 猫艾滋病             置信度：0.14  █ [低]
──────────────────────────────────────────────────
```

### 四种范式结果对比

| 疾病 | P1（HermiT） | P2（Prolog） | P3（Jena） | P4（模糊） |
|------|-------------|-------------|-----------|-----------|
| 猫瘟 | 0.99（SWRL） | 1.00 ✅确诊 | 1.00 ✅确诊 | **0.86 [高]** |
| 犬细小 | 0.77（SWRL） | 0.67 ⚠️疑似 | 0.67 ⚠️疑似 | 不出现（物种过滤） |
| 猫肠炎 | **排除** | **排除** | **排除** | **0.50 [中]** |

**猫肠炎是关键区别**：

- P1-P3：发热命中排除症状 → 完全排除（不在结果中）
- P4：发热严重度高 → 置信度降为中等（仍在结果中）。双输入设计下覆盖率=高+强度=高但排除度=有 → 降一级（中），而非完全压低

### 推理链完整追溯

```
① 症状严重度计算（模糊化输入）
   发热: 39.5°C → 0.80（高烧范围）
   呕吐: 多次   → 0.70（中高频）
   腹泻: 水样+暗红 → 0.90（严重，含血色）

② 覆盖率 + 强度 + 排除度计算
   猫瘟   necessary=[发热,呕吐,腹泻] → 覆盖率=3/3=1.0  强度=(0.8+0.7+0.9)/3=0.80
           exclusion=[咳嗽,流鼻涕]  → 排除度=max(0,0)=0.00

   猫肠炎 necessary=[腹泻,呕吐]      → 覆盖率=2/2=1.0  强度=(0.9+0.7)/2=0.80
           exclusion=[发热,咳嗽]    → 排除度=max(0.8,0)=0.80

③ Mamdani 模糊推理（规则触发）
   猫瘟:   覆盖率=高 ∧ 强度=高 ∧ 排除度=无 → 规则11 → 置信度=高
   猫肠炎: 覆盖率=高 ∧ 强度=高 ∧ 排除度=有 → 规则12 → 置信度=中

④ 去模糊化（重心法）
   猫瘟:   置信度=高 → 0.86
   猫肠炎: 置信度=中 → 0.50
```

## 四、与朴素贝叶斯对比：为什么选模糊推理？

朴素贝叶斯也能处理不确定性，但两者有本质区别：

| 维度 | 模糊推理（P4） | 朴素贝叶斯 |
|------|--------------|-----------|
| **知识来源** | 领域专家定义隶属度函数 | 从数据统计学习概率 |
| **推理性质** | 演绎（规则驱动） | 归纳（数据驱动） |
| **可解释性** | 每条规则可追溯 | 概率黑盒 |
| **与本体关系** | 可扩展为 Fuzzy OWL，嵌入本体 | 独立范式，不嵌入本体 |
| **数据需求** | 低（不需要训练集） | 高（需要大量标注病例） |

引用第四章 4.6 节：

> 模糊推理的隶属度函数需要领域专家定义，不是纯数据驱动——这恰好防止了"统计分数伪装成逻辑结论"的问题。

朴素贝叶斯告诉你"P(猫瘟|发热,呕吐,腹泻) = 0.85"，但它解释不了**为什么**是 0.85。模糊推理告诉你"覆盖率高且强度高且排除度低 → 置信度高"，每一步都有明确的规则依据。

## 五、与 LLM 方案对比

| 维度 | LLM 方案 | P4 模糊推理方案 |
|------|----------|---------------|
| 输入 | 「我的猫 39.5°C，吐了好几次，拉肚子带血」 | `symptoms + symptom_details` |
| 输出 | 「可能是猫瘟，建议就医」 | 猫瘟（0.86 [高]）> 猫肠炎（0.50 [中]） |
| 推理链 | ❌ 黑盒 | ✅ 模糊规则可追溯 |
| 严重度感知 | ⚠️ 可能忽略数值细节 | ✅ 体温/频率/类型映射为严重度 |
| 排除语义 | ⚠️ 可能一刀切排除 | ✅ 根据严重度连续降级 |
| 一致性 | ❌ 同一输入可能不同输出 | ✅ 相同输入必定相同输出 |
| 耗时 | 2-5 秒 | <0.1 秒 |

## 六、P1-P4：什么时候用哪个？

| 场景特征 | 推荐范式 | 原因 |
|---------|---------|------|
| 需要分类推理、一致性检查 | P1 OWL | Tableau 算法完备正确 |
| 开放世界更合理（医疗、法律） | P1 OWL | "没记录" ≠ "没有" |
| 需要递归推理 | P2 Prolog | OWL/Jena 递归能力弱 |
| 规则密集（350+ 条） | P2 Prolog | Prolog 规则表达力强 |
| 需要标准查询语言 | P3 Jena | SPARQL 是 W3C 标准 |
| 需要数据持久化 | P3 Jena | 三元组数据库 |
| **症状有严重程度** | **P4 模糊** | 隶属度函数处理连续值 |
| **排除不是一刀切** | **P4 模糊** | 排除症状降低置信度而非完全排除 |
| **需要领域专家知识驱动** | **P4 模糊** | 隶属度函数由专家定义，非数据驱动 |

> **实践中常组合使用**：用 OWL 管理领域概念体系，用 Prolog 执行复杂规则，用 Jena 提供查询服务，用模糊推理处理不确定性信号——确定性推理负责"硬规则"，不确定性推理负责"灰度地带"。

## 八、总结

P1-P4 构成了"多范式推理实战营"的前四部曲——**确定性推理（P1-P3）+ 不确定性推理（P4）**：

| | P1 | P2 | P3 | P4 |
|---|---|---|---|---|
| 逻辑 | 描述逻辑 | Horn 子句 | 前向链规则 | **模糊逻辑** |
| 假设 | OWA | CWA | CWA | **模糊（0-1）** |
| 输入 | 二元（有/无） | 二元（有/无） | 二元（有/无） | **连续（0-1）** |
| 输出 | 二元（确诊/排除） | 二元（确诊/排除） | 二元（确诊/排除） | **连续（0-1）** |
| 排除 | 完全排除 | 完全排除 | 完全排除 | **连续降级** |
| 外部依赖 | JVM + 补丁 | SWI-Prolog | Docker | **无（纯 Python）** |

> 「确定性推理负责'法律的条文'，不确定性推理负责'法官的自由裁量'。」
